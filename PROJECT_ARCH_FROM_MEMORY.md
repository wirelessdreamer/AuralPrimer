# AuralPrimer Project Architecture (From Memory)

Date written: 2026-03-03  
This is a reconstruction from troubleshooting history, not a live filesystem scan.

## 1. Expected repository shape

The project was a Node monorepo with workspaces similar to:

- `apps/desktop` (Tauri desktop app, gameplay-focused)
- `apps/studio` (Tauri studio app, ingest/transcription-focused)
- `python/ingest` (Python sidecar: `aural_ingest`)
- `packages/*` and `visualizers/*` (shared libs/visual modules)

## 2. Runtime architecture

### Desktop app

- TypeScript frontend in `apps/desktop/src/main.ts`.
- Gameplay rendering:
  - Drums highway canvas.
  - Per-player fretted canvases (tab/Nashville modes).
- Loads MIDI charts and features generated by sidecar.

### Studio app

- TypeScript frontend for import/configure workflows.
- Rust Tauri backend invoked sidecar and streamed progress.
- Exposed algorithm controls (drum filter, melodic method).

### Python sidecar

- Built into a single exe for deployment.
- Performed:
  - source decode/mix
  - stem separation
  - drum transcription
  - melodic transcription
  - SongPack artifact generation

## 3. Data and handoff contracts

### Input

- audio files or stem directories (`import` / `import-dir`)
- optional DTX import path (`import-dtx`)

### Output contract

- `manifest.json`
- `audio/*`
- `charts/notes.mid`
- `features/{beats,tempo_map,sections,events}.json`

Frontend gameplay depended primarily on `charts/notes.mid` + timing/features.

## 4. Build/deploy model

### Sidecar build

- Python deps installed.
- PyInstaller onefile build generated `aural_ingest.exe`.
- Exe copied to both desktop/studio `src-tauri/binaries` locations.

### App builds

- Tauri builds for desktop and studio.
- Portable packaging script assembled:
  - `AuralPrimer.exe`
  - `AuralStudio.exe`
  - `resources/`
  - latest sidecar exe
  - preserved `data/` folder

## 5. Known fragile points (important)

1. Stale sidecar in portable folder can invalidate all UI-level fixes.
2. Wrong/legacy `--drum-filter` values can silently degrade to adaptive fallback.
3. Chart parser strict-vs-relaxed logic can accidentally include melodic tracks if not guarded.
4. Basic Pitch model assets must be bundled and resolvable in frozen runtime.

## 6. Most important engineering rule learned

For transcription regressions, verify runtime binary freshness first (portable sidecar), then analyze algorithm output. UI defaults alone are not reliable proof of behavior.

## 8. Data layout and runtime paths (remembered)

Non-portable:

- settings typically under `%APPDATA%\\dev.auralprimer.app\\settings.json`
- songs under app data `songs/`

Portable mode:

- if executable-adjacent `data/` exists, app resolves data/config there
- portable package expected:
  - `AuralPrimer.exe`
  - `AuralStudio.exe`
  - sidecar exe in portable root
  - `data/songs/*.songpack`

## 9. SongPack minimal fields that mattered most

1. `manifest.json`:
   - `song_id`, `title`, `artist`, `duration_sec`
2. `charts/notes.mid`:
   - primary gameplay source for drums and melodic lanes
3. `features/*.json`:
   - beats/tempo/sections/events support timing and UI overlays

When triaging playback/chart issues, `notes.mid` integrity was usually the fastest truth source.

## 10. Known architectural coupling points

1. Drum visual quality depended on two layers:
   - ingest note generation quality
   - chart parser strict/relaxed selection behavior
2. Keys availability depended on:
   - stem detection aliases (`keys/piano/keyboard/synth`)
   - melodic transcription path generating a keys track
3. Regressions were often cross-layer:
   - parser fix alone cannot recover missing notes never written by ingest
   - ingest fix alone can be hidden by stale sidecar shipping/runtime mismatch
